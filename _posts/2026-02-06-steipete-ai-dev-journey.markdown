[steipete.me](https://steipete.me)는 Peter Steinberger의 블로그야. iOS 프레임워크 PSPDFKit을 13년간 만들고 매각한 시니어 개발자인데, 2025년부터 AI 코딩에 완전히 빠져들었어. 92편의 글 중 AI 개발 관련 핵심을 정리했어.

---

## 누구인가?

Peter Steinberger는 오스트리아 비엔나 출신 개발자야. PDF 뷰어 프레임워크 PSPDFKit을 만들어서 13년간 운영하고 매각했어. 그 후 3년간 심한 번아웃을 겪었는데, 2025년 AI 코딩을 발견하면서 다시 불꽃을 찾았어.

> "행복과 목적은 발견하는 게 아니라 **창조하는 것**이다. 오랫동안 어떤 기술도 이렇게 흥분시키지 못했다. 우리는 돌아왔다. 이제 만들 시간이다."

---

## 핵심 워크플로우: 병렬 Claude 세션

Peter의 핵심 전략은 **3~8개의 Claude Code 세션을 동시에 돌리는 것**이야.

```
리팩토링 → 1~2개 에이전트
테스트/UI/정리 → 약 4개 에이전트
전체 개발 → 최대 8개 에이전트
```

### 어떻게 관리해?

- **Ghostty 터미널** 사용 — VS Code 내장 터미널 대신 독립 터미널
- **Dell 40인치 모니터** — 4개 Claude 세션을 한 화면에 표시
- **탭 색상 구분** — worktree마다 다른 색
- **VibeTunnel** — 각 Claude 세션의 현재 작업 내용을 자동 표시

터미널 탭이 전부 "claude"로 표시되는 문제를 해결하려고 직접 도구를 만들었을 정도야. `vt title "디버깅 중 - 인증 모듈"`처럼 각 세션에 이름을 붙여.

## "Claude Code는 내 컴퓨터다"

Peter는 Claude Code를 **코딩뿐 아니라 컴퓨터의 모든 것**에 사용해:

| 용도 | 예시 |
|------|------|
| 콘텐츠 변환 | Jekyll에서 MDX로 40개 포스트 변환 (20분) |
| 기능 추출 | Swift 프로젝트 생성, 테스트, 오픈소스 공개까지 자동화 |
| 블로그 작성 | 글 작성, 테스트 데이터 생성 |
| 코드 배포 | 커밋, PR 작성, CI 모니터링, 빌드 실패 자동 수정 |
| 시스템 관리 | macOS 설정 변경, 새 머신 설정 |

`--dangerously-skip-permissions` 플래그로 권한 확인을 건너뛰고, 하루 약 1시간을 절약한다고 해. 2개월간 문제가 없었다고 주장하지만, 위험할 수 있으니 주의해야 해.

## 도구 선택 철학: CLI > MCP

Peter는 **MCP 서버 대부분을 제거**했어. 이유는:

> "CLI가 있는 서비스가 훨씬 컨텍스트 효율적이다. MCP는 대부분 CLI로 충분하고, 오히려 컨텍스트를 낭비한다."

실제로 쓰는 도구:
- `gh` — GitHub CLI
- `vercel` — 배포
- `psql` — 데이터베이스
- `axiom` — 로그 분석

MCP 중 유일하게 유용하다고 평가한 건 `chrome-devtools-mcp`뿐이야.

다만 직접 만든 MCP도 있어:
- **Peekaboo** — 스크린샷 찍고 AI로 분석
- **Terminator** — 외부 터미널 관리
- **Conduit** — 고급 파일 조작

## 시간의 20%는 리팩토링

Peter는 개발 시간의 **20%를 리팩토링에 투자**해. 이 작업도 전부 에이전트가 수행해:

- `jscpd` — 중복 코드 감지
- `knip` — 사용하지 않는 코드 제거
- 린터 규칙 적용
- 테스트 개선

> "코드를 계속 깨끗하게 유지하지 않으면 AI도 점점 엉뚱한 코드를 생성하기 시작한다."

---

## CLAUDE.md: 약 800줄의 규칙집

Peter의 에이전트 설정 파일은 약 800줄이야. 포함 내용:

- 제품 설명
- 코딩 패턴 및 규칙
- React 선호사항
- 테스트 방법론
- 슬래시 명령어 (`/commit`, `/automerge`, `/review`)

Claude가 실수하면 "CLAUDE.md를 업데이트해서 이 실수를 다시 하지 않게 해"라고 시켜. 규칙집이 점점 성장하면서 에러율이 줄어든대.

---

## MCP 도구 만들 때의 모범 사례

Peter는 여러 MCP 도구를 직접 만들면서 쌓은 모범 사례도 공유했어:

- **환경 변수는 모두 기본값이 있어야** 해 — 사용자가 추가 설정 없이 시작 가능
- **단일 파일은 500줄 이하** (300줄 이하 권장)
- **매개변수 파싱은 관대하게** — `project_path`를 정의했어도 `path`도 받아들여
- **오류 발생 시 복구 방법을 같이 알려줘** — 에러 메시지만 던지지 말고
- **`stdio`로 출력하면 안 돼** — MCP 클라이언트를 방해하니까 파일 기반 로깅 사용
- **릴리스 전 자동화된 체크리스트** — Git 상태, 보안 감사, 바이너리 검증, 패키지 크기 확인

## "Just One More Prompt" — AI 중독 경고

Peter는 AI 개발의 **어두운 면**도 솔직하게 다뤄:

> "나는 Claudoholic(Claude 중독자)이다."

AI 도구가 너무 강력해지면서 생긴 문제들:

- **FOMO** — "이걸로 수년간 미뤄둔 모든 아이디어를 구현할 수 있다"는 깨달음
- **과도한 업무** — 깨어있는 시간 대부분을 AI 코딩에 투자
- **업계 전반의 과로 문화** — AI 스타트업에서 주 80시간 이상이 표준화

해결책으로 Claude 세션 시간을 상태 바에 표시해서 **얼마나 오래 작업했는지 의식하게** 만들었어.

이 문제를 공유하기 위해 **"Claude Code Anonymous"**라는 밋업도 만들었어. 런던에서 시작해서 비엔나, 베를린, 샌프란시스코 등으로 퍼졌어. "AI 도구 사용으로 인한 수면 부족"을 농담처럼 이야기하지만, 실제 건강 문제를 다루는 진지한 자리야.

## 모델 변천사

Peter의 모델 선택은 시간에 따라 변했어:

| 시기 | 주력 모델 | 이유 |
|------|---------|------|
| 2025년 초 | Claude Code (Sonnet/Opus) | 터미널 기반, 자율적 |
| 2025년 중반 | Claude + Gemini + Cursor 혼합 | 각 모델의 강점 활용 |
| 2025년 말 | GPT-5 Codex 위주 | 더 큰 컨텍스트(230k), 효율적 토큰 사용 |

> "GPT-5 Codex는 더 내향적인 엔지니어 같아. 조용하지만 일관되게 작업을 수행한다."

하지만 Opus의 깊은 사고력은 여전히 높이 평가해. 도구는 바뀌어도 **워크플로우 원칙은 동일**해.

## "바이브 코딩"의 미래

Peter는 바이브 코딩(vibe coding)의 진화를 목격하면서 이렇게 말해:

> "더 이상 코드를 읽지 않는다. 스트림을 지켜보고 핵심 부분만 확인한다. 아키텍처 이해보다 언어/생태계 선택과 의존성 관리가 더 중요해졌다."

> "대부분의 소프트웨어는 복잡한 사고가 필요 없다. 한 형식의 데이터를 다른 형식으로 옮기고, 저장하고, 표시하는 것이다."

실제 사례: 이전에 실패했던 Zig 언어 변환 작업을 단 두 문장 프롬프트로 5시간 만에 완성했어.

## 핵심 교훈 정리

| 교훈 | 설명 |
|------|------|
| **병렬화가 최고의 생산성 비법** | 하나를 최적화하지 말고 여러 개를 동시에 |
| **CLI > MCP** | 대부분의 외부 서비스는 CLI로 충분 |
| **20% 리팩토링** | 코드를 깨끗하게 유지해야 AI도 좋은 코드를 생성 |
| **CLAUDE.md는 살아있는 문서** | 실수할 때마다 규칙 추가 |
| **건강을 지켜** | AI가 강력할수록 과로의 유혹도 커져 |
| **도구는 변해도 원칙은 동일** | 모델이 바뀌어도 병렬화, 검증, 깨끗한 코드 유지 |

Peter의 블로그에서 가장 인상적인 건, 기술적 팁뿐만 아니라 **AI 시대 개발자의 정신 건강**까지 솔직하게 다룬다는 점이야. "우리는 돌아왔다"면서도 "건강하지 못한 수준"이라고 경고하는 균형 잡힌 시각이 돋보여.
