[steipete.me](https://steipete.me)라는 블로그가 있어. Peter Steinberger라는 개발자가 운영하는데, 이 사람이 AI 코딩에 완전 미쳐있어. 2025년에만 37편의 글을 쓰면서 자기가 발견한 모든 꿀팁을 공유했어.

이 글에서는 92편의 글 속에 숨겨진 **진짜 쓸모있는 꿀팁들**을 쉽게 풀어서 설명해줄게.

## 이 사람은 누구야?

Peter Steinberger는 오스트리아 비엔나에 사는 개발자야. **PSPDFKit**이라는 PDF 뷰어 프레임워크를 만들어서 13년 동안 회사를 운영했어. 회사에 직원이 100명이 넘었는데, 결국 회사를 팔았어.

그런데 회사를 팔고 나니까 마음이 텅 비어버렸대. 13년 동안 회사가 곧 자기 자신이었거든. 파티도 가보고, 상담도 받고, 다른 나라에 가서 살아보기도 했는데 아무것도 마음을 채워주지 못했어.

그러다 2025년에 AI 코딩 도구를 발견했어. 컴퓨터 앞에 앉아서 코딩을 시작하니까 **"아, 내 불꽃이 돌아왔다!"**라는 느낌이 들었대.

> "행복과 목적은 찾는 게 아니라 **만드는 것**이다. 오랫동안 어떤 기술도 이렇게 설레게 하지 못했다."

그 뒤로 매일 16시간씩 AI 코딩을 하면서 블로그에 꿀팁을 공유하기 시작했어.

## 꿀팁 1: 한 번에 여러 AI를 동시에 돌려

Peter가 가장 강조하는 꿀팁이야. **Claude Code를 한 개만 쓰지 말고, 3~8개를 동시에 돌리라는 거야.**

이게 무슨 말이냐면, 상상해봐. 숙제가 5개 있어. 혼자 하나씩 하면 5시간 걸리잖아? 그런데 로봇 친구가 5명 있으면? 각자 하나씩 맡기면 1시간이면 끝나!

Peter는 이걸 이렇게 나눠:

```
간단한 리팩토링 → AI 1~2개 동시에
테스트, UI, 코드 정리 → AI 4개 동시에
본격적인 개발 → AI 최대 8개 동시에
```

### 근데 어떻게 8개를 관리해?

여기서 두 번째 꿀팁이 나와. **Git worktree**라는 걸 써.

보통 프로젝트 폴더가 하나잖아? Git worktree를 쓰면 **같은 프로젝트의 복사본 폴더를 여러 개** 만들 수 있어. 각 폴더에서 Claude를 하나씩 돌리는 거야.

```bash
# worktree 사이를 빠르게 이동하는 별칭 설정
alias za="cd ~/worktrees/a"
alias zb="cd ~/worktrees/b"
alias zc="cd ~/worktrees/c"
```

`za`만 치면 A 작업 폴더로, `zb`만 치면 B 작업 폴더로 이동해. 각 폴더에서 Claude가 열심히 일하고 있어.

### 문제: 터미널 탭이 다 똑같아 보여!

Claude를 8개 돌리면 터미널 탭이 전부 "claude"라고 표시돼. 어느 게 어느 작업인지 모르겠어!

Peter는 이 문제가 너무 불편해서 아예 **VibeTunnel**이라는 도구를 직접 만들었어. 각 Claude 세션에 이름표를 붙여주는 거야:

```
vt title "인증 모듈 디버깅 중"
vt title "테스트 작성 중 - PR #456"
vt title "CI 실패 수정 중"
```

이렇게 하면 탭을 보기만 해도 각 AI가 뭘 하고 있는지 바로 알 수 있어.

### 모니터도 중요해

Peter는 **Dell 40인치 울트라와이드 모니터**를 써. 화면이 넓으니까 Claude 4개 창을 한 화면에 다 볼 수 있거든. 각 창의 탭 색깔을 다르게 해서 구분하기 쉽게 만들었어.

## 꿀팁 2: VS Code 터미널을 버리고 Ghostty를 써

Peter는 원래 VS Code 안에 내장된 터미널을 썼는데, 불안정하다고 느꼈대. 그래서 **Ghostty**라는 독립 터미널 앱으로 바꿨어.

Ghostty가 좋은 이유:
- **동기화 렌더링** — 화면이 깜빡이지 않아
- **24비트 색상** — 코드 색상이 예쁘게 표시돼
- **유니코드 지원** — 한국어 같은 특수 문자도 깨지지 않아
- **빠름** — VS Code 터미널보다 훨씬 가벼워

터미널이 뭔지 모르는 친구를 위해 설명하면, 까만 화면에 글자만 나오는 프로그램이야. 마우스 대신 키보드로 컴퓨터에게 명령하는 곳. Claude Code는 이 터미널에서 돌아가거든.

## 꿀팁 3: Claude에게 "모든 것"을 시켜

Peter는 블로그에 "Claude Code는 내 컴퓨터다"라는 제목의 글을 쓸 정도야. 코딩만 시키는 게 아니라 **컴퓨터로 하는 거의 모든 일**을 Claude에게 시켜.

### 블로그 글 40개를 20분 만에 이사

Peter는 블로그 시스템을 Jekyll에서 다른 시스템으로 바꿔야 했어. 글이 40개나 돼. 직접 하면 며칠 걸릴 일이야.

Claude에게 시켰더니 **20분 만에** 40개를 다 변환했어.

### 새 컴퓨터 설정

새 Mac을 사면 이것저것 설치하고 설정해야 하잖아? 이것도 Claude에게 시키면 돼. "내 개발 환경 설정해줘"하면 필요한 프로그램 설치, 터미널 설정, Git 설정까지 다 해줘.

### 코드 배포도 자동으로

코드를 다 짜면 GitHub에 올리고, PR(Pull Request)을 만들고, CI(자동 테스트)가 돌아가는지 확인하고, 실패하면 고치고... 이런 반복 작업도 전부 Claude에게 맡겨.

### 어떻게 가능해?

`--dangerously-skip-permissions`라는 옵션을 켜면 Claude가 매번 "이거 해도 돼요?"라고 물어보지 않아. 하루에 약 1시간을 절약한대. 2개월 동안 써봤는데 문제가 없었다고 해.

**하지만 주의!** 이 옵션은 이름 그대로 "위험하게 권한을 건너뛰는 것"이야. Claude가 실수로 중요한 파일을 삭제할 수도 있어. 초보자는 쓰지 않는 게 좋아. Peter처럼 경험 많은 개발자가 자기 컴퓨터에서 쓰는 거야.

## 꿀팁 4: MCP보다 CLI를 써

MCP는 Claude에게 외부 서비스를 연결해주는 기능이야. 예를 들어 Notion MCP를 연결하면 Claude가 Notion 문서를 읽을 수 있어.

그런데 Peter는 **MCP 대부분을 제거**했어!

> "CLI가 있는 서비스는 CLI를 쓰는 게 훨씬 효율적이야. MCP는 컨텍스트(작업 공간)를 낭비해."

CLI가 뭐냐면, 터미널에서 명령어로 서비스를 이용하는 거야. 예를 들어:

```bash
gh issue list          # GitHub 이슈 목록 보기
gh pr create           # Pull Request 만들기
vercel deploy          # 웹사이트 배포하기
psql -c "SELECT ..."   # 데이터베이스 조회하기
```

이렇게 CLI 도구를 쓰면 Claude가 터미널 명령으로 바로 실행할 수 있어. MCP를 설정하고 연결하는 것보다 훨씬 간단하고, 작업 공간도 아낄 수 있어.

Peter가 유일하게 MCP로 남겨둔 건 `chrome-devtools-mcp`뿐이야. 브라우저 개발자 도구를 AI가 볼 수 있게 해주는 건데, 이건 CLI로 대체하기 어렵거든.

### 그런데 직접 만든 MCP도 있어

Peter가 CLI를 좋아한다고 MCP를 아예 안 쓰는 건 아니야. CLI로 할 수 없는 특별한 경우에는 직접 MCP를 만들어서 써:

- **Peekaboo** — 화면 스크린샷을 찍어서 AI에게 보여줘. "이 버튼이 왜 삐뚤어졌지?" 같은 UI 문제를 해결할 때 유용해.
- **Terminator** — Claude가 다른 터미널 창을 제어할 수 있게 해줘. 여러 Claude를 관리하는 "보스 Claude"를 만들 수 있어.
- **Conduit** — 파일을 고급스럽게 다루는 도구.

## 꿀팁 5: 새로운 프로젝트를 이해하는 "투 컨텍스트 기법"

새로운 코드를 처음 만날 때, Peter만의 독특한 방법이 있어. **"투 컨텍스트(Two Context) 기법"**이라고 불러.

### 1단계: 코드를 텍스트로 변환

**repo2txt**라는 도구에 GitHub 저장소 주소를 넣으면, 프로젝트의 모든 코드를 하나의 텍스트 파일로 변환해줘. 테스트 파일이나 이미지는 빼고 핵심 소스 코드만 뽑아.

### 2단계: AI에게 물어보기

이 텍스트 파일을 Google AI Studio의 Gemini에 넣고 물어봐:

```
이 프로젝트에서 주목할 점은 뭐야?
```

왜 Gemini를 쓸까? Gemini는 작업 공간이 엄청 크거든. 아무리 큰 프로젝트라도 한 번에 다 넣을 수 있어.

### 3단계: 설계 문서(SDD) 만들기

아이디어를 AI에게 말하면 **소프트웨어 설계 문서**로 바꿔줘. 근데 여기서 핵심 꿀팁이 나와!

첫 번째 AI가 만든 설계서를 **두 번째 AI에게 검토**시키는 거야:

```
이 설계 문서에서 부족하거나 이상한 점 20가지를 찾아줘.
```

두 번째 AI가 찾은 문제점을 다시 첫 번째 AI에게 가져가서 수정해. 이걸 3~5번 반복하면 **약 500줄짜리 꼼꼼한 설계서**가 완성돼.

### 4단계: Claude에게 실행 시키기

이 설계서를 Claude Code에게 주면서 "이거 만들어"라고 하면 2~4시간 동안 자동으로 코드를 짜줘.

**마치 선생님 두 명이 서로 숙제를 검사하는 것 같지?** 한 선생님이 쓰고, 다른 선생님이 틀린 곳을 찾고, 다시 고치고... 이렇게 하면 훨씬 완성도 높은 결과가 나와.

## 꿀팁 6: 시간의 20%는 청소에 투자해

Peter의 규칙: **개발 시간의 20%는 코드 청소(리팩토링)에 쓴다.**

이게 왜 중요하냐면, AI가 코드를 짤 때 기존 코드를 참고하거든. 기존 코드가 지저분하면 AI도 지저분한 코드를 생성해. 반대로 깨끗하면 AI도 깨끗한 코드를 만들어.

> "코드를 깨끗하게 유지하지 않으면 AI도 점점 엉뚱한 코드를 생성하기 시작한다."

코드 청소도 AI에게 시켜! 청소에 사용하는 도구들:

### 중복 코드 찾기: jscpd

```
> jscpd를 실행해서 중복된 코드를 찾고, 공통 함수로 합쳐줘.
```

같은 코드가 여러 곳에 복사되어 있으면 찾아서 하나로 합쳐줘. 복사된 코드가 많을수록 AI도 헷갈려하거든.

### 안 쓰는 코드 지우기: knip

```
> knip을 실행해서 사용하지 않는 코드를 찾고 삭제해줘.
```

예전에 쓰다가 이제 안 쓰는 함수나 변수가 남아있으면 지워줘. 코드가 가벼워지고, AI가 읽을 양도 줄어들어.

### 규칙 검사: 린터(Linter)

린터는 "문법 검사기"야. 맞춤법 검사처럼 코드 규칙을 검사해줘. 틀린 부분을 Claude에게 고치라고 시키면 돼.

## 꿀팁 7: CLAUDE.md를 "살아있는 규칙집"으로 만들어

Peter의 CLAUDE.md(에이전트 설정 파일)는 **약 800줄**이야. 엄청 길지?

처음부터 800줄을 쓴 게 아니야. Claude가 실수할 때마다 조금씩 추가한 거야. 핵심 꿀팁은 이거야:

### Claude가 실수하면 "규칙을 스스로 쓰게 해"

```
> 방금 한 실수를 다시 하지 않도록 CLAUDE.md에 규칙을 추가해.
```

이렇게 하면 Claude가 스스로 자기 규칙을 쓰거든. 규칙집이 점점 커지면서 같은 실수를 안 하게 돼. 마치 "오답 노트"를 AI가 알아서 쓰는 거야.

### Peter의 CLAUDE.md에 들어있는 것들

```markdown
# 제품 설명
이 프로젝트는 ~300k LOC의 TypeScript React 앱이야...

# 코딩 패턴
- ES 모듈 사용
- React 컴포넌트는 함수형으로
- 상태 관리는 Zustand 사용

# 테스트 방법
- 기능 완성 후 테스트 작성
- vitest 사용

# 슬래시 명령어
/commit — 커밋 메시지 정리
/automerge — PR 자동 병합
/review — 코드 리뷰
```

### 작업별 메모 폴더도 만들어

특정 작업에 대한 노트는 별도 폴더에 저장하고, CLAUDE.md에서 참조해:

```markdown
인증 관련 작업은 @notes/auth-notes.md 참조
DB 마이그레이션은 @notes/migration-notes.md 참조
```

이렇게 하면 CLAUDE.md가 너무 길어지지 않으면서도 필요한 정보를 찾을 수 있어.

## 꿀팁 8: 프롬프트는 짧게, 스크린샷은 자주

Peter가 AI에게 말하는 방식이 독특해. **프롬프트를 아주 짧게** 써.

> "1~2문장이면 충분하다."

긴 설명보다 **스크린샷 한 장**이 훨씬 효과적이야.

### 예시 비교

**안 좋은 방법 (긴 설명):**
```
로그인 화면의 이메일 입력 필드가 오른쪽으로 치우쳐 있고,
비밀번호 필드와 간격이 너무 넓고, 버튼 색상이 디자인과 다릅니다.
이 세 가지를 모두 수정해주세요.
```

**좋은 방법 (스크린샷 + 짧은 말):**
```
[스크린샷 붙여넣기] 이거 고쳐.
```

AI가 스크린샷을 보면 문제를 한눈에 파악해. 말로 설명하는 것보다 정확하고 빠르지.

### 음성 입력도 활용해

Peter는 **Wispr Flow**라는 음성 입력 도구를 추천해. Mac에서 `fn` 키를 두 번 누르면 음성 입력이 시작돼.

왜 음성이 좋냐면, 타이핑할 때는 귀찮아서 짧게 쓰게 되잖아. 근데 말로 하면 자연스럽게 **더 자세하고 풍부한 설명**이 나와. AI에게는 자세한 설명이 더 좋거든.

## 꿀팁 9: "Blast Radius"를 생각해

"Blast Radius"는 "폭발 반경"이라는 뜻이야. Peter가 AI에게 일을 시킬 때 항상 생각하는 거야.

**이 변경이 얼마나 넓은 범위에 영향을 줄까?**

파일 하나만 바꾸는 건 폭발 반경이 작아 — 잘못돼도 그 파일만 되돌리면 돼. 하지만 데이터베이스 구조를 바꾸는 건 폭발 반경이 커 — 모든 곳에 영향을 줘.

Peter의 규칙:
- **폭발 반경이 작은 작업** → AI에게 맡기고 딴 거 해
- **폭발 반경이 큰 작업** → 계획을 세우고 직접 확인하면서 진행
- **예상보다 오래 걸리면** → 중단하고 상태를 확인

작은 변화를 자주 커밋하면 뭔가 잘못됐을 때 쉽게 되돌릴 수 있어. 한 번에 크게 바꾸지 말고, 작은 조각으로 나눠서 하는 게 안전해.

## 꿀팁 10: Feature 브랜치를 쓰지 않아

보통 개발자들은 새 기능을 만들 때 "feature 브랜치"를 만들어서 작업하잖아? Peter는 안 그래. **main 브랜치에서 직접 작업해.**

이게 가능한 이유:
- 커밋을 아주 작은 단위로 자주 해 (원자적 커밋)
- 문제가 생기면 바로 되돌릴 수 있어
- 브랜치를 합치는 복잡한 과정이 사라져

이건 혼자 작업할 때의 이야기야. 팀으로 일하면 브랜치를 쓰는 게 맞아!

## 꿀팁 11: AI 모델마다 잘하는 게 달라

Peter는 하나의 AI만 쓰지 않아. **상황에 따라 다른 AI를 골라 써.**

| 상황 | 추천 AI | 이유 |
|------|---------|------|
| 일반 코딩 | Claude Code (Sonnet) | 빠르고 똑똑해 |
| 복잡한 문제 | Claude (Opus) | 깊이 생각하는 능력이 뛰어나 |
| 큰 프로젝트 분석 | Gemini | 작업 공간이 엄청 넓어 |
| 디버깅 | o3 | 복잡한 버그를 잘 잡아 |
| 코드 리뷰 | GPT-5 | "내향적인 엔지니어"처럼 꼼꼼해 |

> "GPT-5 Codex는 내향적인 엔지니어 같아. 조용하지만 일관되게 작업을 수행한다."

도구는 계속 바뀌어도 **병렬 작업, 검증, 깨끗한 코드 유지** 같은 원칙은 똑같이 적용돼.

## 꿀팁 12: 구독은 투자야, 아끼지 마

Peter는 AI 도구에 **월 약 800달러(100만원 이상)**를 쓴대. 많다고? Peter의 생각은 달라:

> "시간은 유일하게 다시 충전할 수 없는 자원이다. AI 구독 비용을 투덜거리는 건, 시간당 200달러 벌면서 커피 값 아끼는 것과 같다."

### 돈 절약 꿀팁

그래도 현실적으로 비용이 부담된다면:

- **Claude Max ($200/월)** — Peter가 "최고의 가성비"라고 평가. 거의 무제한 사용 가능.
- **개발자 파트너 프로그램** — API 비용 30% 할인 받을 수 있어
- **Cursor Pro ($20/월)** — 추가 요청당 $0.04. 3시간 코딩해도 추가 비용 약 $8
- **토큰 가격은 계속 내려가** — 2년간 1000배 하락했어. 경쟁이 심해질수록 더 싸져

### 자체 호스팅은 비현실적

Peter가 직접 AI 서버를 돌려봤어. GPU 8개를 빌려서 돌리면 **월 11,000달러**. 그냥 구독하는 게 훨씬 싸.

## 꿀팁 13: MCP 도구를 직접 만들 때의 규칙

Peter는 여러 MCP 도구를 직접 만들면서 쌓은 노하우를 공유했어. MCP 도구를 만들 계획이 있다면 참고해:

### 사용하기 쉽게 만들어

- **환경 변수는 다 기본값을 줘** — 사용자가 아무것도 설정하지 않아도 바로 쓸 수 있어야 해
- **매개변수를 넓게 받아들여** — `project_path`를 기대하는데 사용자가 `path`라고 보내도 받아들여야 해
- **오류 메시지에 해결 방법도 같이 써** — "에러 발생!" 대신 "에러 발생! OO을 설치하면 해결됩니다"

### 코드 품질 지켜

- **파일 하나가 500줄이 넘으면 안 돼** (300줄 이하가 좋아)
- **`stdio`로 출력하면 안 돼** — MCP 통신을 방해하거든. 대신 파일에 로그를 써
- **모든 도구에 `info` 명령을 넣어** — 버전, 의존성 상태, 설정 문제를 보여주는 진단 기능

### 릴리스 전 체크리스트

Peter는 릴리스 전에 자동으로 확인하는 스크립트를 만들었어:
- Git 상태가 깨끗한지
- 보안 취약점은 없는지
- 바이너리가 제대로 빌드됐는지
- 패키지 크기가 2MB를 넘지 않는지
- 베타 태그로 먼저 테스트 후 정식 릴리스

## 꿀팁 14: "그냥 대화해" — 오버엔지니어링하지 마

Peter의 가장 최근 글 제목이 **"Just Talk To It"**이야. 직역하면 "그냥 얘기해."

> "RAG, subagent, 복잡한 도구에 시간 낭비하지 말고 그냥 대화하세요."

Peter의 메시지는 명확해:
- 너무 복잡한 설정에 시간 쓰지 마
- Claude에게 **직접 말해서** 시키는 게 가장 빠르고 효과적
- 어려운 도구를 배우느라 시간 쓰지 말고 **바로 시작해**

특히 초보자에게 중요한 메시지야. 완벽한 설정을 만들어야 한다는 부담 없이, 그냥 Claude를 열고 말하면 돼.

## 꿀팁 15: 라이브 코딩으로 배워

Peter는 자기가 일하는 모습을 **라이브 스트리밍**으로 공개했어. 실제로 1시간 안에 새 기능(Arena — Twitter 사용자 호환성 분석 도구)을 처음부터 끝까지 만드는 과정을 보여줬어.

이 라이브에서 보여준 실전 팁들:

- **에이전트 컨텍스트를 깔끔하게 유지** — HTML보다 마크다운을 선호해서 토큰을 아껴
- **로그를 그대로 복사해서 AI에게 줘** — "이 에러 고쳐"하고 로그를 붙여넣기만 하면 돼
- **테스트는 기능 완성 후에 작성** — 기능이 돌아가는지 먼저 확인하고, 그 다음에 테스트를 추가
- **명확한 주석을 코드에 추가** — AI가 나중에 코드를 읽을 때 의도를 이해하기 쉽게

## 경고: AI 코딩 중독을 조심해

Peter는 꿀팁만 공유하지 않아. **AI 코딩의 어두운 면**도 솔직하게 이야기해.

### "프로그래머를 위한 슬롯머신"

Peter는 AI 에이전트를 **슬롯머신**에 비유했어.

슬롯머신은 버튼을 누르면 결과가 나오잖아? 대박이 날 수도 있고, 꽝일 수도 있고. 그 **예측할 수 없는 결과**가 뇌에서 도파민을 만들어내서 계속 하고 싶게 만들어.

AI 코딩도 비슷해. 프롬프트를 입력하면 놀라운 결과가 나올 수도 있고, 엉망이 될 수도 있어. 그 불확실한 기대감이 **중독적**이야.

### "나는 Claude 중독자다"

Peter는 런던에서 열린 밋업에서 자신을 "Claudoholic(클로드 중독자)"이라고 소개했어.

증상들:
- 깨어있는 시간 대부분을 AI 코딩에 써
- 수년간 미뤄둔 아이디어를 전부 구현하고 싶은 충동
- "프롬프트 하나만 더..." 하면서 잠을 미루기
- 삶에서 AI 코딩 외의 것이 사라짐

> "이 도구들은 여가 시간을 침해하고 있다. 솔직히 '삶이 없다'는 걸 인정한다."

### 업계 전체가 과로 중

Peter만의 문제가 아니야:
- AI 스타트업에서 **주 80시간 이상** 일하는 게 표준화되고 있어
- Google 공동 창립자 Sergey Brin은 **주 60시간**이 최적이라고 말했어
- Cognition CEO는 "주말 포함 극단적 성과 문화"를 유지한대

### Peter의 대응

Peter는 자기가 이 함정에 빠졌다는 걸 인식하고, 몇 가지 대책을 만들었어:

1. **Claude 세션 시간을 상태 바에 표시** — 얼마나 오래 했는지 의식하게
2. **"Claude Code Anonymous" 밋업** 창설 — 런던에서 시작해서 비엔나, 베를린, 샌프란시스코 등 여러 도시로 확산. 개발자들이 모여서 경험을 공유하는 안전한 공간
3. **격렬한 업무 ↔ 완전한 휴식** 주기를 반복

밋업 형식도 재밌어: "나는 OO인데, Claude Code가 XX했을 때..."로 시작하는 5분짜리 발표를 하는 거야. 비공개라서 녹화도 안 해.

## 정리: Peter의 핵심 철학

| 원칙 | 설명 |
|------|------|
| **그냥 대화해** | 복잡한 설정보다 직접 말하는 게 빠르다 |
| **병렬이 왕이다** | AI 1개를 잘 쓰는 것보다 8개를 동시에 돌리는 게 낫다 |
| **CLI > MCP** | 외부 서비스는 CLI로 연결하는 게 더 효율적 |
| **작은 커밋, 자주** | 큰 변경보다 작은 변경을 자주 하면 안전하다 |
| **코드를 깨끗하게** | AI도 깨끗한 코드에서 깨끗한 코드를 만든다 |
| **실수는 규칙으로** | Claude가 틀릴 때마다 CLAUDE.md에 규칙 추가 |
| **투 컨텍스트** | AI 두 개가 서로 검토하면 완성도가 올라간다 |
| **스크린샷 활용** | 긴 설명보다 화면 캡처 한 장이 낫다 |
| **건강을 지켜** | AI가 강력해질수록 과로의 유혹도 커진다 |

Peter의 블로그가 특별한 이유는, 꿀팁만 나열하는 게 아니라 **실패와 중독, 번아웃까지 솔직하게 다루는 점**이야. "우리는 돌아왔다. 만들 시간이다"라면서도 "건강하지 못한 수준이다"라고 경고하는 균형 잡힌 시각이 돋보여.

AI 코딩은 정말 강력한 도구야. 하지만 도구는 도구일 뿐이야. **도구를 잘 쓰는 것만큼 도구에 휘둘리지 않는 것도 중요해.**
