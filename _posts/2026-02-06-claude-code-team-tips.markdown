Claude Code를 만든 Boris Cherny가 **Claude Code 팀이 실제로 쓰는 방법**을 공유했어. 팀원마다 쓰는 방식이 다르지만, 공통적으로 효과가 입증된 10가지 팁이야.

"Claude Code를 쓰는 올바른 방법은 하나가 아니야 — 각자의 설정은 다 달라. 뭐가 잘 맞는지 직접 실험해봐!"라고 했어.

---

## 1. 병렬 작업이 가장 큰 생산성 향상

Git worktree를 3~5개 동시에 띄우고, 각각에 Claude 세션을 하나씩 돌려. 이게 **가장 큰 생산성 향상 비법**이야.

```bash
# worktree 사이를 빠르게 이동하는 별칭
alias za="cd ~/worktrees/a"
alias zb="cd ~/worktrees/b"
alias zc="cd ~/worktrees/c"
```

각 worktree는 완전히 분리된 작업 디렉토리라서 작업끼리 서로 섞이지 않아. 어떤 팀원은 로그 분석 전용 worktree를 따로 유지하기도 해.

핵심 원리: **하나의 똑똑한 세션을 최적화하는 것보다 여러 세션을 병렬로 돌리는 게 더 효과적**이야.

---

## 2. 막히면 다시 계획 모드로

Plan Mode는 처음에만 쓰는 게 아니야. **중간에 막혔을 때 복구하는 도구**이기도 해.

진행이 안 되면 코드를 계속 수정하려 하지 말고, Plan Mode로 돌아가서 다시 생각해.

한 팀원의 워크플로우:
1. **Claude A**에게 계획을 작성하게 해
2. **Claude B**를 새로 열어서 그 계획을 "시니어 엔지니어처럼 검토해"라고 시켜
3. 검토를 반영한 후에 구현 시작

계획이 잘 세워지면 Claude가 한 번에 구현을 끝내는 경우가 많아.

---

## 3. Claude가 스스로 규칙을 쓰게 해

Claude가 실수할 때마다 이렇게 말해:

```
> CLAUDE.md를 업데이트해서 이 실수를 다시 하지 않게 해.
```

Claude는 자기 규칙을 쓰는 걸 잘해. 팀은 이 "살아있는 규칙집"을 계속 다듬으면서, 에러 발생률이 눈에 띄게 줄었어.

어떤 팀원은 작업별 노트 디렉토리를 만들고 CLAUDE.md에서 그 노트를 참조하게 해서, 축적된 경험을 재사용해.

---

## 4. 반복 작업은 Skill로 만들어

하루에 한 번 이상 하는 일이 있으면 Skill로 만들어서 버전 관리에 넣어.

팀에서 실제로 쓰는 Skill 예시:

- **`/techdebt`** — 세션 끝에 중복된 코드를 찾아서 제거
- **컨텍스트 덤프** — Slack, Google Drive, Asana, GitHub의 최근 7일 내용을 한 곳에 모아줘
- **분석 에이전트** — dbt 모델을 만들고, 코드 리뷰하고, 개발 환경에서 테스트까지

Skill을 버전 관리에 넣으면 **새로 합류한 팀원이 바로 검증된 워크플로우를 물려받아**. 조직의 지식이 자동으로 전달되는 셈이야.

---

## 5. Claude가 스스로 버그를 고치게 해

Slack MCP를 연결하면 버그 스레드를 바로 붙여넣고 "고쳐"라고 할 수 있어.

```
> 이 버그 스레드를 보고 원인을 찾아서 수정해.
```

Claude에게 **방법을 세세하게 지시하지 않아도 돼**. 문제만 알려주고 접근 방식은 Claude에게 맡겨. Claude는 로그의 잡음 속에서 패턴을 찾아 분산 시스템 문제를 해결하는 걸 잘해.

CI 테스트가 실패해도 마찬가지야. 에러 로그를 주고 "고쳐"라고 하면 자동으로 수정해.

---

## 6. Claude를 동료처럼 대해

Claude를 지시하는 대신 **도전하면** 더 좋은 결과가 나와:

```
> 이 변경사항을 빡세게 검토해. 네 테스트를 통과할 때까지 PR 만들지 마.
```

```
> 이게 작동한다는 걸 증명해봐.
```

그리고 평범한 수정이 나왔을 때:

```
> 지금까지 알게 된 모든 것을 바탕으로, 이걸 버리고 우아한 해결책을 구현해.
```

Claude를 "시키는 대상"이 아니라 **"설득해야 할 동료"**처럼 대하면 결과물의 품질이 올라가.

---

## 7. 터미널 환경을 꾸며

터미널 설정도 생산성에 영향을 줘:

- **Ghostty** 터미널 추천 — 동기화 렌더링, 24비트 색상, 유니코드 지원
- **`/statusline`** — 컨텍스트 사용량과 git 브랜치를 상태 바에 표시
- **탭 색상 구분** — worktree마다 다른 색으로 구분
- **음성 받아쓰기** — macOS에서 `fn` 두 번 누르면 음성 입력 가능

음성 받아쓰기는 의외로 강력해. 타이핑보다 **더 자세하고 자연스러운 프롬프트**가 나오거든. 길게 설명하는 게 귀찮을 때 말로 하면 훨씬 편해.

---

## 8. Subagent로 컨텍스트를 깨끗하게

Subagent를 쓰는 세 가지 패턴:

**패턴 1: 작업 분산**
프롬프트에 "subagent를 사용해"를 추가하면 계산이 많은 작업을 분산시킬 수 있어.

**패턴 2: 메인 컨텍스트 보존**
개별 작업을 subagent에게 보내서 메인 에이전트의 작업 공간을 깨끗하게 유지해.

**패턴 3: 보안 검증**
권한 요청을 Opus 모델의 subagent에게 보내서 보안 스캔과 자동 승인을 시켜.

---

## 9. SQL을 직접 쓸 필요 없어

데이터베이스 CLI(BigQuery, Postgres, SQLite)를 Skill로 만들어두면:

1. Claude가 쿼리를 작성하고
2. Skill이 쿼리를 실행하고
3. Claude가 결과를 해석해줘

데이터베이스 도구로 왔다갔다하는 대신 **하나의 컨텍스트에서 전부 처리**할 수 있어. Boris는 6개월 동안 SQL을 직접 작성한 적이 없다고 해.

팀 전체가 Claude Code 안에서 분석 쿼리를 돌려. 데이터 분석이 개발 워크플로우의 일부가 된 거야.

---

## 10. Claude로 배우기

Claude Code를 **학습 도구**로도 활용할 수 있어:

- **`/config`에서 "Explanatory" 또는 "Learning" 출력 스타일 활성화** — Claude가 추론 과정을 설명해줘
- **HTML 프레젠테이션 생성** — 낯선 코드베이스를 설명하는 슬라이드를 만들어줘
- **ASCII 다이어그램** — 프로토콜이나 아키텍처를 그림으로 이해
- **간격 반복 학습** — Claude가 나중에 다시 질문해서 내가 제대로 이해했는지 확인해줘

코드를 읽는 것만으로는 이해하기 어려운 부분도, Claude에게 "초등학생한테 설명하듯이 알려줘"라고 하면 확 와닿게 설명해줘.

## 숨겨진 공통 패턴

팀원마다 방식은 다르지만, 공통적인 원리가 있어:

| 원리 | 설명 |
|------|------|
| **최적화보다 병렬화** | 하나를 잘 쓰는 것보다 여러 개를 동시에 |
| **반복적 계획** | Plan Mode는 시작뿐 아니라 복구에도 |
| **자기 개선** | Claude가 스스로 실수 패턴을 기록하게 |
| **Skill 축적** | 반복하는 건 자동화 |
| **동료 관계** | 지시하지 말고 설득하고 도전하기 |

## 정리하면

Boris가 강조한 가장 중요한 메시지: **"Claude Code를 쓰는 올바른 방법은 하나가 아니야."**

팀의 다양한 접근 방식이 수렴하는 세 가지 공통 원칙이 있어:

1. **검증** — 항상 결과를 확인하게 해
2. **병렬화** — 여러 세션을 동시에 돌려
3. **적절한 위임** — AI에게 맡길 수 있는 건 맡겨

처음 시작한다면 **병렬 worktree**(팁 1)와 **CLAUDE.md 자동 업데이트**(팁 3)부터 해봐. 이 두 가지만으로도 체감이 크게 달라질 거야!
